When learning software architecture in Go, one topic that looks simple at first—but quickly gets subtle—is **database transactions with the repository pattern**.

At some point, you’ll ask yourself:

> If transactions belong to the persistence layer, why do they often appear in the service layer?
> Isn’t that crossing responsibilities?

This question sits right at the intersection of clean architecture, repository pattern design, and Go’s explicit approach to database handling. The answer is not just about code — it’s about **where business consistency actually lives**.

Let’s break it down.

---

## The Need for Atomic Operations in Applications

Almost every real-world application has operations that must succeed or fail **as a unit**:

* placing an order
* creating a user and provisioning resources
* transferring money
* reserving inventory

These operations typically span **multiple repositories**:

```text
Create Order
 ├─ OrderRepository
 ├─ InventoryRepository
 └─ PaymentRepository
```

From a business perspective, this is **one use case**.
From a database perspective, this must be **one transaction**.

This is where the design tension starts:

* Transactions come from the database layer
* Atomicity belongs to business logic

So who should own it?

---

## Common Patterns (and Where They Start to Hurt)

### Repository Owns the Transaction

A common first attempt is letting each repository manage its own transaction.

```go
func (r *OrderRepository) Create(ctx context.Context, order Order) error {
    tx, err := r.db.BeginTx(ctx, nil)
    if err != nil {
        return err
    }
    defer tx.Rollback()

    _, err = tx.ExecContext(ctx, "INSERT INTO orders ...")
    if err != nil {
        return err
    }

    return tx.Commit()
}
```

This looks clean and encapsulated. The repository handles persistence concerns internally.

But problems appear immediately when multiple repositories must coordinate:

```go
orderRepo.Create(...)
inventoryRepo.Reserve(...)
paymentRepo.Create(...)
```

Each repository commits independently.
If the third call fails, your system is now in an inconsistent state.

What looked like good encapsulation becomes **fragmented atomicity**.

---

### Service Layer Owns `*sql.Tx`

The next approach is letting the service explicitly control the transaction:

```go
tx, _ := db.BeginTx(ctx, nil)
defer tx.Rollback()

orderRepo.Create(ctx, tx, order)
inventoryRepo.Reserve(ctx, tx, items)
paymentRepo.Create(ctx, tx, payment)

tx.Commit()
```

This solves consistency, but introduces a new discomfort:

* service now depends on `*sql.Tx`
* persistence detail leaks into application layer
* repository signatures become noisy

You gain correctness, but lose architectural clarity.

---

### Hidden Transactions via Context

Some designs hide transactions inside `context.Context`:

```go
ctx = context.WithValue(ctx, txKey, tx)
```

Repositories pull the transaction from context if present.

This reduces explicit dependencies, but introduces hidden behavior:

* difficult to reason about
* harder to test
* implicit coupling
* surprises during debugging

This pattern often trades **explicitness for magic**, and Go usually prefers the opposite.

---

## Patterns That Matter More Than Library Choice

Before choosing a specific implementation, a few principles matter more than the mechanics.

### Transaction scope belongs to the use case

Atomicity is a **business decision**, not just a persistence detail.

The service layer should decide:

> These operations must succeed or fail together.

It should not decide:

* isolation level
* SQL statements
* lock strategy
* retry behavior

That distinction keeps responsibilities clean.

---

### Repositories should not commit business flows

Repositories represent **data access**, not business workflows.

When a repository commits, it assumes it knows the transaction boundary — which it usually doesn’t.

This leads to partial updates and inconsistent state.

---

### Transaction mechanics should stay in infrastructure

Even if the service defines atomicity, it should not care about:

* `sql.Tx`
* `BEGIN/COMMIT`
* driver-specific APIs

This suggests introducing an abstraction.

---

## The Sweet Spot: Transaction Manager / Unit of Work

The balanced approach is:

* Service defines **atomic use case**
* Infrastructure handles **transaction implementation**
* Repositories remain **transaction-agnostic**

This usually looks like a transaction manager.

### Transaction Manager Interface

```go
type TransactionManager interface {
    WithinTransaction(ctx context.Context, fn func(ctx context.Context) error) error
}
```

Infrastructure implementation:

```go
type SQLTxManager struct {
    db *sql.DB
}

func (m *SQLTxManager) WithinTransaction(
    ctx context.Context,
    fn func(ctx context.Context) error,
) error {
    tx, err := m.db.BeginTx(ctx, nil)
    if err != nil {
        return err
    }

    defer tx.Rollback()

    ctx = context.WithValue(ctx, txKey{}, tx)

    if err := fn(ctx); err != nil {
        return err
    }

    return tx.Commit()
}
```

Service layer:

```go
func (s *OrderService) PlaceOrder(ctx context.Context, cmd PlaceOrder) error {
    return s.txManager.WithinTransaction(ctx, func(ctx context.Context) error {
        if err := s.orderRepo.Create(ctx, cmd.Order); err != nil {
            return err
        }

        if err := s.inventoryRepo.Reserve(ctx, cmd.Items); err != nil {
            return err
        }

        return s.paymentRepo.Create(ctx, cmd.Payment)
    })
}
```

Now the service says only:

> Run this use case atomically.

It does not know how transactions are implemented.

---

## Making Repositories Transaction-Aware Without Coupling

Repositories can read the transaction from context and fallback to DB.

```go
func (r *OrderRepository) exec(ctx context.Context) DBTX {
    if tx, ok := ctx.Value(txKey{}).(*sql.Tx); ok {
        return tx
    }
    return r.db
}
```

Then:

```go
func (r *OrderRepository) Create(ctx context.Context, order Order) error {
    db := r.exec(ctx)

    _, err := db.ExecContext(ctx, `
        INSERT INTO orders (id, total)
        VALUES (?, ?)
    `, order.ID, order.Total)

    return err
}
```

Repositories stay clean and reusable.

---

## Advantages of This Balance

This approach gives you:

* clear transaction boundaries
* atomic business operations
* minimal leakage of persistence details
* reusable repositories
* easier testing
* clean architecture separation

Most importantly, it avoids pushing responsibility too far in either direction.

---

## Gotchas (Don’t Overdo It)

Even good abstractions can become over-engineered.

### Avoid full-blown Unit-of-Work frameworks early

You don’t need:

* generic repository factories
* nested transaction graphs
* lifecycle registries
* dynamic repository binding

Start simple. A small transaction manager is enough.

---

### Keep transactions short

Do not:

* call external APIs
* publish messages
* wait for queues
* perform long computations

inside a transaction.

Transactions are for **database consistency**, not orchestration.

---

### Avoid nested transactions unless necessary

Most drivers don’t support true nested transactions.
Savepoints exist, but they add complexity and should be used intentionally.

---

### Don’t hide everything

Too much abstraction can make debugging painful.
If developers can’t tell whether code runs inside a transaction, the abstraction went too far.

---

## Conclusion

Transactions originate from the persistence layer, but **transaction boundaries belong to business use cases**.

The cleanest design separates these concerns:

* Service layer defines atomicity
* Infrastructure implements transactions
* Repositories remain transaction-agnostic

A small transaction manager provides the balance between correctness and architectural clarity.

In Go — where explicitness is preferred — this approach keeps code readable, predictable, and easy to reason about.

And perhaps the most important takeaway:

> The service layer should not know *how* transactions work,
> but it should decide *when* they are needed.
